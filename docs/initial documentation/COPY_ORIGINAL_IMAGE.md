# Copy original image — context-menu architecture

## Контекст

В проекте применены оптимизации загрузки: для отображения в Output / истории / диалоге используются уменьшенные `mid_*` версии изображений вместо оригиналов. Это критично для производительности, особенно в сетке истории.

Однако нативное контекстное меню Chrome ("Копировать изображение") копирует ровно те байты, на которые указывает `<img src>` — то есть `mid_*`. Пользователю это давало неожиданно поджатую картинку при вставке.

## Решение

Перехватываем правый клик в `ZoomableImage` (компонент внутри `components/image-dialog.tsx`) и показываем своё контекстное меню. Загрузка оригинала происходит **только по явному действию пользователя** — оптимизации полностью сохранены.

## Компоненты решения

### 1. `<img src>` остаётся mid

Никаких автоматических префетчей оригинала. `previewSrc` указывает на `currentEntry.outputUrl` (mid), как и до правки. Это значит:
- Открытие диалога так же быстро, как раньше.
- Переключение стрелками между изображениями не качает оригиналы.
- Память и трафик на уровне до изменений.

### 2. Перехват правого клика

`ZoomableImage` принимает новые пропсы:
- `originalUrl` — URL полноразмерной версии (приходит из `effectiveDownloadUrl`).
- `downloadFilename` — имя файла для пункта "Скачать".

Обработчик `handleContextMenu` вызывает `e.preventDefault()` (чтобы Chrome не показал своё меню) и сохраняет позицию курсора в state `menuPos`. Меню рендерится `position: fixed` с clamp по viewport, чтобы не вылазило за край.

### 3. Три действия меню

| Пункт | Что делает | Когда грузит оригинал |
|---|---|---|
| **Копировать изображение** | fetch → canvas → PNG → `navigator.clipboard.write` | в момент клика |
| **Скачать** | fetch → blob → `<a download>` | в момент клика |
| **Открыть в новой вкладке** | `window.open(originalUrl)` | браузер, не мы |

### 4. Почему конвертация в PNG через canvas

`ClipboardItem` нативно поддерживает только `image/png` во всех современных браузерах. JPEG/WebP/AVIF blob либо выбрасывает `NotAllowedError` в Chrome, либо молча падает. Поэтому:

1. `fetch(originalUrl)` → `Blob` (в исходном формате).
2. `createImageBitmap(blob)` → декодирование.
3. `canvas.drawImage(bitmap)` → растеризация.
4. `canvas.toBlob('image/png')` → пере-кодирование в PNG.
5. `clipboard.write([new ClipboardItem({ 'image/png': pngBlob })])`.

Стоимость: ~100–300 мс на типичной картинке + временное удвоение памяти на время конвертации (canvas + bitmap). Универсально работает во всех браузерах.

### 5. Состояния копирования

State `copyState`: `idle | loading | done | error`. Текст пункта меню меняется соответственно ("Копирование…", "Скопировано ✓", "Ошибка копирования"). После `done` меню закрывается через 600 мс, после `error` — через 1200 мс. Кнопка "Копировать" заблокирована во время `loading`.

### 6. Закрытие меню

Эффект на `menuPos` подписывает три глобальных слушателя (только пока меню открыто):
- `mousedown` вне меню → закрыть.
- `Escape` → закрыть.
- `scroll` (capture) → закрыть (иначе меню "висит" в воздухе).

Атрибут `data-image-context-menu` на корне меню используется для `closest()` — клики внутри не считаются "outside".

## Что НЕ меняли (и почему)

### Миниатюры в истории

В сетке истории нативное меню Chrome остаётся. Альтернативы рассматривались и отклонены:

- **Кастомное меню на каждой миниатюре** — добавляет 60+ обработчиков, и пользователю всё равно нужно открыть просмотр чтобы убедиться какую именно миниатюру он копирует. Лишний слой UX без выгоды.
- **Hover-префетч** — частичный фоновой трафик при наведении, сводит на нет смысл оптимизаций.

Текущий маршрут для копирования оригинала из истории: клик по миниатюре → диалог → правый клик → "Копировать изображение". Один лишний клик, нулевой урон оптимизациям.

### Кнопка "Скачать" под картинкой

Кнопка `handleDownload` в самом диалоге не тронута — она и так грузила оригинал по `effectiveDownloadUrl`. Меню дублирует её для удобства правого клика.

## Требования окружения

- **Secure context** (https или localhost) — `navigator.clipboard.write` не работает на http.
- **User activation** — клик пользователя по пункту меню как раз даёт нужную активацию, всё ок.
- **CORS** — оригиналы должны отдаваться с CORS-разрешением (иначе `createImageBitmap`/canvas получит tainted source). На текущем бекенде проблем нет; если появятся — `copyState` встанет в `error` и пользователь это увидит.

## Файлы

- `components/image-dialog.tsx` — единственное место, где живёт логика. `ZoomableImage` получил пропсы `originalUrl` / `downloadFilename`, новый блок state + хелперов копирования / скачивания / открытия, `onContextMenu` на контейнер, и JSX меню.

## Drag-to-new-tab → оригинал

Отдельная микро-правка в том же файле: на wrapper-div внутри `<DialogTrigger>` повешен `onDragStart`, который перехватывает native drag-and-drop и подменяет URL в `DataTransfer`:

```ts
onDragStart={(e) => {
  const orig = entry.originalUrl ?? downloadUrl ?? entry.outputUrl;
  if (orig && orig !== entry.outputUrl) {
    e.dataTransfer.setData("text/uri-list", orig);
    e.dataTransfer.setData("text/plain", orig);
  }
}}
```

**Как это работает.** Когда пользователь хватает миниатюру `<img>`, браузер автоматически кладёт `src` в `DataTransfer` под MIME `text/uri-list` — этот URL используется при drop'е на таб-бар / адресную строку. `dragstart` всплывает до wrapper-div'а, и React'овский хендлер срабатывает синхронно до фактического drop'а — `setData` переписывает браузерные дефолты. `text/plain` пишется вторым как fallback для платформ, где таб-бар читает plain text.

**Почему это не ломает оптимизации.** Мы кладём в `DataTransfer` только *строку*, не байты. Оригинал загрузится только если пользователь реально дропнет ccылку и браузер по ней перейдёт — ровно то же, что произошло бы при нажатии «Открыть в новой вкладке» в контекстном меню. Никакого фонового трафика.

**Покрытие.** Логика живёт в `ImageDialog`, поэтому работает везде, где этот компонент используется: Output today, sidebar history, все будущие места. Без правок в отдельных файлах.
